Ollama Desktop - Help

Welcome! This application is a Graphical User Interface (GUI) for Ollama. It allows you to comfortably control locally installed AI models, fine-tune parameters, and enforce structured JSON responses.

🚀 Quickstart

  1. Start the Ollama server in the background.
    (Terminal: ollama serve)
  2. Click on Get LLM List in the top left of the app.
    The list of your installed models will load.
  3. Select a model from the dropdown menu (e.g., llama3 or gemma2).
  4. Enter your question below and press the Play Button (▶).
Tip: Ensure the address in the top left is correct (Default: 127.0.0.1:11434).

✍️ The Intelligent Prompt Area

The lower section of the app is designed to make working with complex prompts and multiple files as easy as possible.

1. Automatic Resizing

The text box (your input) thinks ahead: It automatically expands when you write or paste longer texts. Once a maximum height is reached, it smoothly transitions into a scrollable area. This ensures your screen space isn't wasted while keeping large code blocks manageable.

2. Attached Files (File Panel)

When you attach files or images to your prompt using the buttons (e.g., + File), they are clearly listed right above the text box. The interface dynamically rearranges the layout so you can always see exactly which context files are currently being sent to the AI model.

3. Quick Controls

  • Shift + Enter: Inserts a standard line break so you can format and structure your text.
  • Clicking the Play Button (▶) or Enter: Sends your complete request (including all attached files) to the model.

🔄 Operating Modes & Vision Support

You can control how the app communicates with the model via the API dropdown menu (top center). There are two modes:

1. Generate Mode (Single Request)

This mode is designed for one-off tasks ("One-Shot").

  • Vision / Images: This is the only mode where you can upload images (Buttons + File or + Screenshot bottom left). Use Vision models like llava or moondream for this.
  • Behavior: Each request stands alone. The model immediately "forgets" the previous question.
  • Context: However, it is possible to include the Context Tokens in the next request. This allows for a conversation even in Generate Mode.

2. Chat Mode (Conversation)

Here, the entire conversation history is saved and sent along with every new message.

  • Model Switching: A highlight of this mode is its flexibility. You can switch the LLM mid-conversation (e.g., from a fast 7B model to a smart 70B model) without losing the thread of the chat.

🔀 The Bridge: "Generate > Chat"

Since the Chat mode cannot receive images directly, the app offers an intelligent workflow:

  1. Select Generate and upload an image (e.g., "Describe this image").
  2. Wait for the model's response.
  3. Now click the Generate > Chat button at the top.

Result: The image analysis (Question + Answer) is copied into the Chat history. You can now seamlessly ask follow-up questions about the image in Chat Mode, as the model now "knows" the context.

🛠 Model Parameters

1. System Prompt

Here you define the "personality" of the AI. Check the Use System Prompt box to send this instruction before every chat.

Example: "You are an experienced C# developer. Answer briefly and precisely."

2. Output Format (JSON Mode)

Especially useful for developers: If you need structured data, you can define a JSON Schema here. The model will be forced to respond exactly in this format.

3. Content Prompt

This prompt expands your input by appending the content of attached files. These are generally files containing text (.txt, .json, .pdf...). Images are handled separately and passed to the LLM via an Image parameter.

4. Options Parameter Details

In the "Options Parameter" table, you can precisely control the AI's response behavior. Here is the meaning of the standard settings:

Option Name Description
temperature Creativity & Predictability.
Controls how "daring" the model's choices are.
0.0: Deterministic (always the same response).
0.7 - 0.8: Balanced, natural responses (Default).
1.2+: Very experimental, diverse, or even chaotic.
top_p Nucleus Sampling.
Only considers words whose cumulative probability reaches the value P.
Example: 0.9 means the model only considers words that make up the top 90% of the probability mass. Good for preventing completely absurd answers.
top_k Selection Limit.
Strictly limits the selection of the next word to the K most likely words.
Example: 40 only allows the top 40 candidates. Lower values make the model more focused and grammatically stable, but less varied.
num_ctx Context Window (Input + Output).
Determines how many tokens (word pieces) the model can process at once.
1024: Low VRAM usage, only for short chats.
2048: Standard for simple tasks.
4096 - 8192: Recommended for document analysis.
Note: Higher values massively increase VRAM consumption.
num_keep Permanent Context.
Determines how many tokens from the beginning (usually the System Prompt) are kept in cache when the context is full.
Example: 24 ensures core instructions aren't forgotten when the model starts deleting old messages.
num_predict Maximum Response Length.
Limits the number of tokens to generate.
Example: 100 equals roughly 75 words.
-1: Unlimited (until stop token).
-2: Fills the remaining space in the context window.
seed Random Number Generator Seed.
A fixed value (e.g., 42) ensures that the model always delivers the exact same response given the exact same parameters and prompt. Useful for debugging and reproducibility.
min_p Minimum Probability.
Alternative to top_p. A word must have a minimum probability relative to the most likely word.
Example: 0.05 removes unlikely "outliers" often more efficiently and naturally than top_p.
typical_p Linguistic Naturalness.
Selects tokens that are "typical" for the context. Helps preserve natural language by balancing out mathematically highly probable but contextually repetitive words. (Default usually 1.0).
repeat_last_n Repetition Protection Range.
Determines how many tokens the model looks back to avoid repetitions.
Example: 64 checks the last 64 tokens. 0 disables the check.
repeat_penalty Repetition Penalty.
Penalizes words that have already appeared.
1.0: Neutral.
1.1 - 1.2: Good balance to prevent getting "stuck" in word loops.
presence_penalty Topic Diversity.
Penalizes tokens that are already present in the text (regardless of their frequency). Increases the likelihood of the model talking about new aspects or topics.
frequency_penalty Vocabulary Variation.
Penalizes tokens based on how often they have already appeared. Effectively prevents the model from using the same sentences or specific phrases over and over again.
tfs_z Tail Free Sampling.
Reduces the influence of very unlikely words at the end of the probability list ("Tail"). Similar to top_p, but weights the structure of the probability distribution more heavily.
width Image Width.
The width of the generated image in pixels.
Should ideally match the model (e.g., 512 for SD 1.5, 1024 for SDXL/Flux).
Note: Values should be divisible by 8 or 64.
height Image Height.
The height of the generated image in pixels.
Together with the width, this determines the Aspect Ratio.
Higher values require significantly more VRAM.
num_steps Denoising Steps.
Number of iterations to develop the image from noise.
1-10: Fast, but often blurry or flawed.
20-30: Good standard for quality and speed.
50+: Higher details, but diminishing returns with double the wait time.
cfg_scale CFG Scale (Prompt Adherence).
Determines how strictly the model adheres to your text prompt.
< 5: Very creative, the model "dreams" more freely.
7.0 - 8.0: Standard value. Good balance between prompt and image quality.
> 15: Forces the model rigidly into the prompt, often leads to artifacts or "oversharpened" images.
seed Reproducibility (Seed).
The starting value for the initial image noise.
Same Seed + Same Prompt + Same Settings = Exactly the same image.
Use this to modify an image slightly (e.g., change color) without completely rerolling the composition.

➕ Expand the List

Missing a parameter? This list is not rigid! Ollama and new models often support additional parameters.

You can simply click on the bottom empty row (marked with an asterisk *) in the table and enter your own parameter name as well as its value. As long as the loaded model supports this parameter, it will be applied.

⏱ Timeout Settings

In the bottom left, you will find the Timeout [s] setting.

  • The default value is often set to 600 seconds (10 minutes).
  • If you are using very large models or generating very long texts, timeouts may occur. Increase this value if necessary.

🛠 Tools & Function Calling

The Tools tab turns the LLM into an agent that can actively perform tasks (e.g., checking weather, performing calculations). The entire process runs automatically.

1. Tool System Prompt

This is the "stage direction" for the model. It is used in two phases:

  1. Decision: The model uses this prompt to understand when to call a tool.
  2. Processing: After the tool is executed, the model uses the User Prompt and the Model System Prompt to correctly integrate the tool's output into the response.
Important: Check the Use Tool box in the top right to activate this function.

2. Tool JSON (Schema)

Here you define the interface (API) of your tool in JSON format. The model reads this JSON schema to know which parameters it needs to extract from the user's request (e.g., "city": "Berlin").

3. Tool Python Code

Here you store the logic that should be executed. This code is automatically executed by the app as soon as the model requests the tool.

The Workflow:

  • The model detects a tool requirement and provides JSON parameters.
  • The app executes your Python code with these parameters.
  • The return value of the Python code is automatically integrated into the original prompt.
  • The model generates the final response based on your tool's result.
Note: The Python code runs locally. Ensure all required libraries (e.g., requests) are installed in your Python environment. You can copy the Python code's output to the clipboard using the Tools resp > CLP button.

📄 RAG Tool (Chat with Documents / Knowledge Base)

Use this tab to upload your own text files or documents (.txt or .pdf). The content of these files is sent to the model as context so it can answer questions about your specific data.

How it works (The Workflow)

  1. Loading: The selected file is analyzed and internally split into individual sentences.
  2. Analysis: When you ask a question, the LLM first analyzes it and creates a list of search words, synonyms, and abbreviations (based on internal configuration).
  3. Search: The app searches the split sentences for these terms ("Hit Count").
  4. Response: The found text passages are provided to the LLM as background knowledge to precisely answer your question.
Important: Check the Use RAG Tool box in the top right of the tab so your documents are considered in the response.

🎚 The Delta Parameter (Context)

Next to the file selection, you'll find the Delta dropdown menu (0-9). This value controls how much context around a hit is sent to the LLM.

Value Meaning
0 Only the exact sentence where the search word was found is passed. (Very strict, context is often missing).
1 The hit sentence plus 1 sentence before and 1 sentence after are passed. (Recommended Default).
2-9 Increases the context area accordingly. Useful if correlations are spread across long paragraphs.

⚙️ Background Info

The logic for keyword determination ("Keyword Extraction Assistant") is stored in the RAG tool settings. The system intelligently searches not just for your word, but also for logical relatives (e.g., searching for "Car" also finds "Automobile" or "Vehicle").

1. RAG Tool System Prompt

This is the "stage direction" for the model. It is used in two phases:

  1. Decision: The model uses this prompt to understand when to call the RAG tool and how search words should be determined.
  2. Processing: After the RAG tool is executed, the model uses the User Prompt and the Model System Prompt to correctly integrate the filtered text results into the request.

2. RAG Tool JSON (Schema)

Here you define the interface (API) of the RAG tool in JSON format. The model reads this JSON schema to know which parameters it needs to extract from the user request (e.g., "words":["apple pie","recipe","cake","apple","baking"]).

💻 Code Generation & Execution

Ollama Desktop can not only display generated program code, but also execute it directly. This supports languages like Python, PowerShell, Batch, HTML/JavaScript, and more.

1. Execute List (Tab: Code Parameter)

For the app to know how to execute code, interpreters must be defined. In the table, you specify:

Column Meaning
Language The name of the language as it appears in the markdown code block (e.g., python, powershell).
Program The path to the interpreter (e.g., python.exe, node.exe, cmd.exe).
Argument Startup parameters (e.g., /C for Batch or -File for PowerShell).
Tempfile The name of the temporary file where the code is saved prior to execution.

2. Execution (Tab: Code Block)

When the LLM has generated a code block (visible in the Response HTML tab), switch to the Code Block tab. The code is automatically extracted there.

Controls:

  • Language Selection (Dropdown): If the LLM generated multiple code blocks (e.g., first Python, then SQL), select the desired block here.
  • Run: Saves the code into the temporary file and executes it with the configured interpreter. The result appears in the "Output" window.
  • UseShellExecute:
    • ✅ Enabled: The program starts in its own, external window (useful for GUI scripts). The output is not displayed in the app.
    • ⬜ Disabled: The program runs in the background. The text output (console) is intercepted and displayed directly in the lower window of the app.
Security Warning: Only execute code that you understand! AIs can hallucinate and generate code that deletes files or creates infinite loops.

🤖 SSH Bot (Autonomous AI Agent)

The SSH Bot transforms Ollama Desktop into an automated Linux system administrator. You provide a goal, and the AI connects to your server via SSH, executes commands, analyzes their output, and autonomously decides on the next logical steps.

Important: Activate the Use SSH Bot switch at the very top of the tab to ready this mode.

1. Prompts & Instructions

  • SSH Bot System Prompt: Here you define the role and ground rules for the AI (e.g., "You are a highly qualified Linux system administrator...").
  • SSH Bot Response Prompt: Here you define how the response from the Linux server (i.e., the output after an executed command) is formatted and sent back to the LLM. This ensures the AI understands exactly what happened on the server after its last action (e.g., by passing the terminal output) to plan the next logical step.

2. Connection Details

Enter the credentials for your target server here:

  • SSH Host / IP: The network address of your server (e.g., 192.168.0.90).
  • Username & Password: Your login credentials.
  • Management: Use Test to check the connection, and Save SSH / Load SSH to save various server profiles and all other settings.

3. Advanced Parameters & Security

Setting Description
Terminal-Prompt RegEx A regular expression the app uses to detect when the server is ready for new input (e.g., when reimar@PI-OLLAMA-TEST:~ $ appears). Only upon a match does the app know the previous command has finished.
Sudo-Prompt & Password If the AI executes commands with sudo, the app intercepts the password prompt (Sudo-Prompt) and automatically submits the Sudo-Password stored here.
Max. Agent-Steps Security Limit: Limits the maximum number of consecutive steps the bot is allowed to execute for a task (e.g., 15) to avoid expensive infinite loops.
Auto Execute ✅ Enabled: The bot acts fully autonomously and executes generated commands immediately.
⬜ Disabled: The model merely suggests the command, and you must manually trigger the execution.
Re Insert System If enabled, the System Prompt is resent with every single agent step so the model doesn't lose its focus (the Administrator role) over longer tasks.
Security Warning: An autonomous AI agent with SSH access and sudo privileges can make far-reaching changes to the system. Use this function with caution and ideally on isolated test systems (like virtual machines or Raspberry Pis).
